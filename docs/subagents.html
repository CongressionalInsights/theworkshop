Subagents are choreography, not magic.

A good subagent system does one thing cleanly: it breaks work into bounded jobs, hands the right job to the right specialist, and keeps the parent thread responsible for judgment, integration, and final truth. TheWorkshop adds structure around that runtime so the delegation is visible, auditable, and safe to learn from.

3 / TheWorkshop overlay

TheWorkshop adds three disciplined layers.

Native subagents stay the runtime. TheWorkshop adds a planning layer that decides what should run, an observability layer that tells the truth about what actually happened, and a staged learning layer that prevents messy durable writes in the middle of delegated work.

Planning / Runtime / Observability
Planning layer

Project, workstream, and job plans decide ownership, order, and acceptance criteria before execution starts.

  • Canonical job selector: agent_profile
  • Planning metadata: references/agents/*.json
  • Project contract: WORKFLOW.md

4 / Pick the path

The run path changes the closeout contract.

The generic idea is stable: spawn bounded work and keep the parent on synthesis. TheWorkshop then draws a sharper line between manual/external runs, dispatch-driven runs, and loop-driven runs, because each one reaches curation and telemetry differently.

5 / Learn without smearing state

Memory and lessons are staged, not sprayed across the system.

This is where TheWorkshop gets opinionated. Subagents may discover durable rules or reusable lessons, but they should not write project memory or canonical lesson files directly. They stage candidates first. Promotion happens once the run or loop reaches a proper terminal boundary.

6 / Surface map

Files that matter when you reason about TheWorkshop subagents.

Broadly, subagents are about spawn, scope, and integration. In TheWorkshop, the files below show exactly where those responsibilities live, which is why the skill stays auditable instead of becoming a black box.

Runtime surfaces
Project agent config .codex/agents/*.toml Executable repo-local roles such as theworkshop_worker and curator agents.
Shared agent library ~/.codex/agents/*.toml Cross-repo reusable roles that stay generic and narrow.
Planning registry references/agents/*.json Planning metadata, budgets, and defaults. Not the executable runtime.
Runtime settings .codex/config.toml Per-project limits like depth and concurrency that travel with the repo.
Canonical telemetry logs/agents.jsonl Manual and dispatch runs both land here so the dashboard has one truthful event stream.
Manual closeout theworkshop agent-closeout The one terminal path for manual or external runs that should also promote staged learning.
Staged durable memory .theworkshop/memory-proposals/*.json Durable rules are proposed here first, optionally scoped by agent_id.
Staged lessons .theworkshop/lessons-candidates/*.json Reusable heuristics and pitfalls land here before curation into canonical lesson files.
7 / Guardrails

What good subagent usage looks like in practice.

The broad advice and the TheWorkshop advice line up: make delegation explicit, keep ownership crisp, and treat durable learning as a closeout concern instead of a casual side effect.

Do / Avoid
Do
  • Spawn for bounded jobs with clear outputs and acceptance criteria.
  • Keep the parent thread on integration, decisions, and final synthesis.
  • Use repo-scoped agents when the repo has local rules worth encoding.
  • Stage durable memory and lessons first, then promote through curators or explicit closeout.
  • Use agent-log for non-terminal manual telemetry and agent-closeout once at the end.
Avoid
  • Parallel overlapping write ownership just because concurrency is available.
  • Letting a worker silently mutate durable memory or canonical lessons mid-run.
  • Using stale planning metadata as if it were the executable runtime definition.
  • Relying on raw terminal-looking log events to imply a correct closeout.
  • Turning “subagents” into a giant catalog of vague generalists.
